# 審計：這套制度能否讓 Sonnet/Opus/Haiku 達到 Fable 5 的表現？

> 撰寫：2026-07-07，Fable 5 session（此使用者最後一次 Fable 5 額度）。
> 回答「不靠 Fable 5，這套流程夠不夠」。結論先講，證據與缺口在後。怎麼用見 `USAGE.md`。

## 結論（一段版）

**日常工作的大約八成可以達到同等表現——條件是照 `USAGE.md` 的方式用，而不是把檔案放著。**
把「Fable 5 的表現」拆開看：執行品質（實作、搜尋、重構、驗證）靠拆解＋可判定驗收＋
驗證不自驗可以逼近同等；長 session 不失焦靠指揮官不下場＋context 紀律可以逼近同等；
但「第一手拆解模糊需求」「察覺任務前提錯誤」「知道何時該停」這三種判斷只能部分外化，
天花板是 Opus 4.8 ＋ 多方案評審 ＋ 使用者仲裁。這不是制度寫得不夠好——這三種能力
本質上難以規則化，`ops/judgment-rubrics.md`#6 已誠實承認其中的品味部分。

## 逐項評估：Fable 5 的優勢 → 制度的替代 → 可回收程度

| Fable 5 優勢 | 制度的替代方案 | 可回收程度 |
|---|---|---|
| 長 context 不失焦 | 指揮官不下場＋原料不進主對話＋隨做隨寫（`ops/harness-diagnosis.md`#3、#4） | 高——制度設計的甜蜜點 |
| 執行準確度 | 三要素交辦＋可判定驗收＋fresh-context 驗證（`ops/model-dispatch.md`） | 高——Sonnet 在規格明確的任務上差距本就小，驗證迴圈補掉剩餘 |
| 錯了會自己發現 | 換路訊號 checklist（`ops/judgment-rubrics.md`#4）＋升級帶失敗軌跡 | 中——弱模型最缺的能力；checklist 是拐杖不是義肢，需要使用者偶爾抽查 |
| 模糊需求的第一手拆解 | Opus 先出計畫＋使用者在 prompt 端多做拆解（`USAGE.md` 第 3 節） | 中低——**最大殘餘缺口**，見下 |
| 品味／感官判斷 | 實測工具化、A/B 給使用者選（`ops/judgment-rubrics.md`#6） | 低——制度已誠實標示補不了 |

## 弱指揮官的四個實戰失效模式（本次審計發現的缺口與修法）

制度檔寫給 subagent 的部分完整；最脆弱的環節是**主對話的指揮官本身**。
Sonnet 當指揮官時可預期的失效：

1. **規則漂移**：session 中段忘記派工、自己下場掃檔案。CLAUDE.md 常載三規則是第一道防線，
   但沒有 session 中途的重新錨定機制。修法：使用者用觸發句重新錨定（`USAGE.md` 第 3 節）；
   可選強化：UserPromptSubmit hook 注入一行提醒——動 `.claude/settings.json` 需使用者
   自行決定，制度不擅自加。
2. **驗收條件寫得糊**：整套制度的品質天花板就是「指揮官寫的驗收條件」。弱指揮官寫出
   「做好功能 X」等級的驗收，後面的驗證全變形式。修法：重要任務讓使用者先審驗收條件
   再放行（`USAGE.md` 第 3 節「先計畫後執行」句式）——這是整套制度最便宜的人力槓桿。
3. **拆解方向錯**：模糊需求下 Sonnet 的第一刀常切錯，之後執行再好也是在錯的方向上快跑。
   修法：模糊任務先 `/model opus` 出計畫（或兩方案擇一），Sonnet 只執行已確認的計畫
   （`USAGE.md` 第 2 節「貴模型在前、便宜模型在後」）。
4. **失敗回報美化**：「理論上可行」「應該就能動」。`ops/judgment-rubrics.md`#2 有規則，但規則
   防不了不讀規則的模型。真正的防線是驗證永遠 fresh-context（做的人說什麼不算數）——
   它已存在於制度中，關鍵是不要為省 token 跳過它。

## 成本現實（不要期待「一樣好又一樣快」）

用便宜模型＋制度換 Fable 5，代價不是品質而是**輪次與延遲**：多一層派工、多一輪驗證、
偶爾升級重做。以 API 牌價估（2026-06 參考值，訂閱折算未公開；Sonnet 5 牌價 UNVERIFIED）：
Sonnet 4.6 費率約 Fable 5 的 1/3、Haiku 4.5 約 1/10，多花 50% 輪次仍是淨省。時間上每個
任務多 1–3 分鐘派工往返。日常工作划算；「今晚就要、品質要頂」的單次任務直接
`/model opus` 硬上比較快。
