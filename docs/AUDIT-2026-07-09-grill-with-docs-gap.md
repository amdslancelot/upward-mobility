# Audit 2026-07-09 — 能否用 grill-with-docs 補上 Fable5 判斷缺口

## 背景

`project/meta/AUDIT-2026-07-07.md` 結論：`upward` plugin 能補齊約 80% 的 Fable5 日常執行紀律（若照 `USAGE.md` 使用），但留下四項判斷缺口，這些是「沒有外部檢查可靠、純靠判斷力」的部分：

1. 模糊需求的第一刀拆解
2. 抓出任務前提本身錯誤
3. 知道何時算做完
4. 品味／感官判斷（無法用 build/test 驗證的「好不好」）

本次 audit 問題：另一套獨立技能系統（Matt Pocock skills，`~/.agents/skills`）裡的 `grill-with-docs`（訪談式釐清機制，答案存進 CONTEXT.md／ADR）能否補上這四項缺口？

## grill-with-docs 機制摘要

結構化訪談：一次問一題、model 每題先給推薦答案但由使用者決定、可從 code 查到的事實用查不用問、動工前需使用者確認已達成共識、確認後的決策與詞彙固化進 CONTEXT.md／ADR。核心協定明講「動工前」（pre-implementation only）。

## 逐項結論

| 缺口 | 判定 | 理由 |
|---|---|---|
| 1. 模糊需求拆解 | **部分補** | 訪談機制逼出規格，但拆解方式來自使用者作答，不是 model 冷判斷；model 只負責問對問題順序 |
| 2. 抓錯任務前提 | **確實補到，最強一項** | 機制會拿使用者說法對照實際 code／glossary，抓出矛盾——這是 model 真判斷（查證），不是甩鍋。天花板：只抓「跟 code／glossary 衝突」的前提，前提錯但不衝突仍漏抓 |
| 3. 何時算做完 | **完全沒補** | grilling 明講只做動工前協定（"不到使用者確認不動工"），對執行中「這樣算不算做完」零機制 |
| 4. 品味／感官判斷 | **幾乎沒補** | 沒有「做出來→看→調整」迴圈，訪談問不出使用者事前講不清楚的品味；ADR 只能記錄品味決定「做出來之後」，無法幫忙先做出那個決定 |

## 核心判斷：人為介入 vs model 判斷力

grill-with-docs 本質是**範圍縮減技巧**：把「模糊任務」透過盤問使用者轉成「有規格任務」，再把結果凍進 ADR/glossary 讓便宜 model 照規格執行。這是**把判斷轉嫁給使用者**，不是**讓 model 本身判斷力變強**。

若按原始 Fable5 平價問題的框架（「不靠人類介入，sonnet/opus 能不能單靠自己打平 fable5」）——grilling 不算數，因為它的核心機制本身就是人類介入迴圈。

唯一真正算「model 判斷力提升」的部分：靠 code／glossary 對照抓前提錯誤那條（缺口 2）。缺口 3、4 維持原 audit 認定的狀態，沒被補上。

## 後續問題：能否把 grilling 式主動提問整合進 brief 階段（ops-plan 階段〇）

`ops-plan` skill 階段〇已有觸發點：「方向不確定（需求模糊、你在替使用者假設偏好或商業脈絡）→ 停下確認再動工」——但只有「何時該停」，沒有「停下後怎麼問」的結構。grilling 補的正是這個「怎麼問」的程序層，不是新的觸發點。

### 成本考量（把人為介入算進整體效能，不當免費資源）

grilling 原生「relentless、走遍每個決策分支」的姿態，是設計給主動要求 stress-test 的使用者用的。若原封不動搬進 ops-plan 階段〇，會反過來吃掉 upward 整套設計最想省的資源——使用者注意力。

具體失敗情境：使用者說「加個設定頁」，若比照 relentless 逼問，階段〇一次射出 8 題（儲存後端、驗證規則、i18n、auth 範圍、空狀態文案……），其中多數有明顯常規預設，model 本該照 core.md「選常規預設」直接做掉、不必問。使用者為一個該花零輪確認的任務被迫回答 8 輪，長期會不信任 brief 階段、甚至學會跳過它。

### 是否連帶補到缺口 3（何時停）或缺口 4（品味）

不會，也不該指望。缺口 3、4 本質發生在 brief 凍結**之後**（執行中的完工判斷／無法事前言傳的品味），brief 階段問得再細也碰不到還沒出現的執行期判斷。把「brief 問得更周全」等同「補上整個判斷缺口」是範圍蔓延，兩者不同。

### 淨結論與建議設計（若採用，僅限有界形式）

**淨判斷：有限度採用，當現有觸發點的內層程序，不當新關卡。**

若要落地，改動範圍限定在 `plugin/upward/skills/ops-plan/SKILL.md` 階段〇：

- 現有觸發點（方向不確定）維持外層閘門不變，只有它已觸發時才啟動下列內層問法，不當階段〇預設步驟。
- 一次一題，model 每題先給自己的推薦答案。
- **該問／該自己判斷的分野**：有安全常規可套 → 直接採用、寫進 brief 的「假設」欄，不問；只在選擇取決於「只有使用者知道的偏好或商業脈絡」時才問（對齊 core.md 既有「三種情境」第二種）。
- **上限 3 題**：超過代表 brief 本身範圍太模糊，該請使用者重講範圍，而非繼續逼問。
- 可從 repo／glossary 查到的事實一律查，不問使用者。

`core.md` 的「三種情境」規則本身不動，維持外層閘門；本設計只是閘門觸發後、原本「停下確認」這一步的具體程序。
