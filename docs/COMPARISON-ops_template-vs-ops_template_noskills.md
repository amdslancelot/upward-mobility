# 兩版模板包的比較報告：ops_template vs ops_template_noskills

撰寫：2026-07-06，Fable 5 session。

## 實驗背景

同一個 session、同一個模型（Fable 5）、同樣的知識內容，寫了兩包模板：

- `docs/ops_template/`（v1）：在 caveman（極簡口吻）與 ponytail（最簡實作）兩個常駐 plugin 啟用的環境下寫成。
- `docs/ops_template_noskills/`（v2，即本資料夾）：使用者關閉兩個 plugin（normal mode）之後，以自然文風重寫。

兩版的規則、數字、佔位符、檔案結構刻意保持一致，差異應該只來自寫作風格 — 但實驗結果顯示不只如此，見下。

## 量化差異

| 指標 | v1（ops_template）| v2（noskills）| 差異 |
|---|---|---|---|
| 總行數 | 519 | 578 | +11% |
| 總字元數 | 16,720 | 21,219 | **+27%** |
| 檔案數與結構 | 9 檔 | 9 檔，結構相同 | 無差異 |

字元數比行數更能反映真實差異，因為行數被兩版相同的表格與 code block 稀釋。膨脹最多的是 `agent-dispatch-mechanics.md`（69 → 85 行）與 `prompt-templates.md`（94 → 106 行）— 正是論證密度最高的兩份文件。

## 質性差異：多出來的 27% 是什麼

**1. 規則附上了理由。** v1 給指令，v2 給指令加原因。例如「省略必填欄就不要派工」在 v2 變成「省略任何一個必填欄位就先補齊再派工，因為缺欄位的交辦幾乎必然產出需要重做的結果，反而更貴」。對弱模型讀者的實際影響：附理由的規則在邊界情況下比較不會被誤用 — 模型知道規則要保護什麼，就能判斷眼前的情況是否在保護範圍內。

**2. v2「順手」補了 v1 沒有的內容。** 這是事前沒預期到的發現：不受最簡壓力寫作時，作者自然寫出了幾處 v1 完全沒有的澄清。例如：

- `model-dispatch.md`：「SendMessage 續用與驗證不自驗不衝突 — 續用是給連續的做事，驗證永遠新開」。v1 裡這兩條規則並列存在，弱模型可能覺得互相矛盾。
- `prompt-templates.md`：「一次審查只選一個焦點 — 焦點混在一起會讓每種檢查都變淺」的理由句。
- `letter-template.md`：多了一整段「為什麼需要這封信」，解釋交接信補的是「檔案讀不出來的後設資訊」。

換句話說，**壓縮省掉的不只是字，還悄悄砍掉了一些本來會存在的思考**。這是兩個模式對文件品質的真實影響 — 即使它們的邊界規則明文豁免檔案內容。

**3. 表格格位 vs 散文。** v1 把論證塞進表格 cell（碎片句），v2 讓表格只裝事實、論證回到段落。

## 方法學缺陷（讀這份報告前必須知道的）

1. **不是乾淨的 A/B。** 同一作者、同一 session，寫 v2 時已經看過 v1 — 結構與用詞有錨定效應，真實差異可能被低估。乾淨的做法是讓一個沒看過 v1 的 fresh-context agent 從原始 ops 檔獨立重寫（本次未執行）。
2. **v1 本來就不是 caveman 文體。** 兩個模式的邊界規則本就規定檔案內容正常寫，所以量到的 +27% 是模式的殘餘壓力（ponytail 的「最短」哲學滲進文件密度）加上自然變異，不是模式的全部效果。
3. **審查狀態不對等。** v1 經過 fresh-context 對抗審查並修了 6 個 findings；v2 沒有跑過同樣的審查，只被作者自己抓到一個錯字（「能做also是」— 順帶示範了寫更多字等於更多出錯表面）。要正式採用 v2，建議先補跑一輪對抗審查。

## 結論與建議

知識內容相同，選哪包看用途：

- **給弱模型日常照做**：兩版等效，v1 便宜 27%（每次載入的 input token）。
- **預期會遇到很多規則沒涵蓋的邊界情況、或文件要給人讀**：選 v2 — 理由句是邊界推理的原料。
- **不建議兩包都留著維護**：雙倍的腐化面積。挑一包，刪另一包。
