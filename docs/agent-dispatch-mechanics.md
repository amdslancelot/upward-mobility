# 派工決策的機制原理與成本結構

> 通用於 Claude Code 類 harness。實測數字來自一次 2026-07 的 session，量級參考用 — 換環境、換方案先重量再引用。
> 讀者：主對話模型（需要推理「該不該派工」「該不該壓縮」的邊界情況時讀本檔；日常照 `ops/model-dispatch.md` 的表即可）。

## 一、誰在做決定：harness 是水管，模型是唯一的決策者

使用者訊息裡「用 X 模型做這件事」對 CLI 而言只是普通文字 — CLI 不解析、不路由。唯一能讀懂並執行的是主對話模型（由 /model 設定決定）。「主對話模型思考」本身就是一次遠端 API 呼叫，沒有本地推理。CLI 只做機械工作：跑 hooks、組 payload、送 API、渲染輸出、執行 tool call。

## 二、派工決定的機械形式

每個 turn 的每一步是一次主模型 inference。「派工」與「不派工」的差別只是這次 inference 生成的內容：

- 不派 = 輸出純文字。1 次遠端呼叫，turn 結束。
- 派 = 輸出一個 Agent tool_use block。CLI 攔截後對便宜模型發起新呼叫，subagent 自己跑多輪，回報再回灌主模型收尾。至少 3 次遠端呼叫。

「決定不派」不留任何 API 痕跡 — 只表現為「少了一次呼叫」。沒有獨立的路由器或決策模組。

## 三、無狀態 API：「完整呼叫」的解剖

API 無狀態 — 伺服器不保存對話；「模型記得前面」的唯一原因是 CLI 每次把整場對話物理性重送。一次呼叫的 payload：system prompt（含 CLAUDE.md、記憶、hook 注入，約 20–30k）+ 全部 tool schema + 完整對話歷史 + 新增量。一個 turn 內跑 N 個 tool call = N+1 次呼叫，每次重送全部。模型每次呼叫都是失憶重生；上一輪輸出成為下一輪 input。

Cache 的作用：伺服器快取「與上次完全相同的前綴」（預設 TTL 5 分鐘），舊前綴按 cache read 計價（API 牌價約 0.1 倍 input 價），新增部分全價 + 寫入 cache 1.25 倍（5 分鐘 TTL；1 小時 TTL 是 2 倍）。實測一輪（原版專案單次量測，你的環境可能差 2–10 倍）：cache read 113,846 tokens、新 input 76、cache 寫入 2,779、output 987。

每輪成本 = 重送全部（舊的 0.1 倍）+ 新增量全價 + 寫 cache 1.25 倍 + output 全價。單輪不貴；貴的是「新增量 × 之後每輪都變成要重送的歷史」的累積 — **把大量原料放進主 context 的行為在複利計息**。

## 四、成本結構真相（單次 session 實測）

便宜模型派工的固定成本約 20k tokens/次（實測多次 haiku 派工：18–27k），來自冷啟動：subagent 載自己的 system prompt、讀檔重建脈絡、多輪執行。

| 項目 | 不派工 | 成功派工 |
|---|---|---|
| 主模型呼叫次數 | 1 | 2（寫交辦 prompt；讀回報收尾）|
| 主模型 output | 純答案（短）| 交辦 prompt 常更長（含內容與驗收條件）|
| 主模型 input | 完整 context ×1 | 完整 context ×2 |
| 便宜模型 | 0 | 約 20k |

三個推論：

1. 派工不省主模型的 output，常常反而更多。省的是「執行迴圈」：subagent 讀檔、修改、重試的多輪全算便宜模型頭上。
2. cache 省的是「重讀舊 context」，省不了兩樣：**執行輪次的費率差**（主模型自己跑 4 輪 = 4 次完整呼叫全按主模型費率；API 牌價 2026-06 查證參考值：Fable 5 $10/$50、Opus 4.8 $5/$25、Sonnet 4.6 $3/$15、Haiku 4.5 $1/$5 每百萬 token in/out——過期值只用於理解量級，引用前到官方 pricing 頁重查 — 頂級模型與 haiku 約差 10 倍）和**永久性 context 污染**（自己讀的原料永遠留在主 context，之後每輪重付，還提早吃滿 window）。
3. 判準是任務的**執行輪次**不是難度：

| 任務型態 | 正解 | 理由 |
|---|---|---|
| 答案已在主模型 context 裡（指涉、澄清、判斷）| 直接答 | 0 輪執行，派工 = 純付 20k 冷啟動，且交辦 prompt 得先把答案寫一遍 |
| 1–2 輪、產物短 | 兩可，偏直接做 | 省的輪次蓋不過冷啟動 |
| ≥3–4 輪、或要讀大量原料 | 派工 | 費率差 + 污染複利，20k 很快回本 |

## 五、字面指令 vs 意圖：偏離要揭露

使用者指令「用便宜模型做」的意圖是省貴模型額度。當字面照做反而更貴（一句話的澄清題），主模型應照意圖處理 — 但必須在回覆中揭露「沒派、為什麼」，讓使用者能 override。使用者知情後仍堅持字面時，照辦，不再重複論證。

## 六、/compact 與 /clear：context 的重置手段

`/compact`：一次貴呼叫把歷史壓成摘要，之後每輪送「摘要 + 近期訊息」。代價是失真（行號、路徑、原文錯誤、數字常消失）。規則：先存檔再壓；壓在任務邊界不壓進行中；可帶指示 `/compact 保留檔案路徑、行號、未完成事項`；自動 compact 不挑時機，手動優於被動。

`/clear`：歸零，零成本。下一件事不相關時用 — 新 session 從 CLAUDE.md 索引重建，比 compact 摘要便宜且準確。

一句話判準：**同一件事沒做完 → /compact；換一件事 → /clear。** 兩者都是使用者打的指令；模型的義務是在任務邊界提醒。

## 七、token 計量的數據源（模型看不到自己的用量）

1. Session log：`~/.claude/projects/<專案>/<session-id>.jsonl` — 每次 API 呼叫的 input/cache_creation/cache_read/output tokens 與 model 欄位。第三方工具（ccusage 等）都是讀這個檔加總。
2. 內建指令（使用者直接打，不耗模型 token）：`/cost`、`/context`、`/usage`。
3. 派工回報：每個 subagent 收工附 usage 區塊 — 收集這些數字，是校準「20k 冷啟動」在你環境實際值的方法。
4. 權威對帳：使用量儀表板 — 訂閱額度對 cache 與不同模型的折算規則未公開，只有儀表板準。

主對話模型自己看不到自己的計量，要知道就得讀 log 檔或請使用者打 /cost。
