# 審查任務的派工哲學（Fable 5 session 實錄）

> 撰寫：2026-07-06。本檔回答一個具體問題：「要審查一批產出時，該用哪個 agent/skill、什麼 model、帶什麼 prompt」。
> 與 `fable5_agent_dispatch_philosophy.md`（派工的成本機制）互補：那檔講該不該派，本檔講審查這類任務怎麼派。

## 一、先分清審查的類型 — 工具跟著類型走

「review」不是一種任務，是四種。選錯工具的審查會看起來有做、實際沒覆蓋到風險：

| 審查類型 | 在審什麼 | 正確工具 |
|---|---|---|
| Diff/PR 正確性 | 這次改動有沒有 bug | `/code-review` skill 或 cavecrew-reviewer agent（吃 git diff）|
| 過度工程 | 有沒有不該存在的抽象/依賴 | `/ponytail-review` skill（只獵複雜度）|
| 文件一致性 | 跨檔矛盾、佔位符、互引斷鏈、弱模型會誤讀的模糊句 | general-purpose agent + 審查範本（見下）|
| 機械 read-back | 檔案存在？內容完整？句子沒斷？ | general-purpose agent，haiku 即可 |

判準一句話：**審 diff 用 diff 工具，審內容用 general-purpose 帶自訂檢查面向**。專用工具（reviewer agent、review skill）的視角是硬編碼的，超出它視角的風險它一律沉默 — 沉默不等於沒問題。

## 二、fresh-context 不是形式，是測試條件

審查一律新開 agent（不 SendMessage 續用先前任何 agent），兩個理由：

1. **驗證不自驗**：做事者（或看過做事過程的 agent）帶著「作者視角」，會腦補出文件裡沒寫的脈絡，看不見歧義。
2. **冷啟動本身就是模擬讀者**：制度檔和模板的目標讀者就是「不知道前情的弱模型」— 一個 fresh-context agent 讀不懂的地方，未來 session 也讀不懂。它的困惑是數據，不是噪音。

實例：本 session 審查 ops_template 時，fresh-context sonnet 抓到「README 寫五種範本、實際檔案有六種」— 作者（主對話）自己剛加了第六種，視角上「當然知道」，所以自己驗永遠驗不出這種 bug。

## 三、model 選擇：haiku 驗形式，sonnet 驗語義

- **haiku**：checklist 可完全機械化時 — 檔案存在、行數、grep 某字串為 0 hits、段落標題齊全。本 session 的制度檔 read-back 用 haiku（26,973 tokens），夠用。
- **sonnet**：finding 需要推理時 — 「這句有兩種讀法」「這個數字看似通用其實是專案殘留」「這兩條規則在邊界情況下打架」。本 session 的模板審查用 sonnet（38,502 tokens），抓到的高嚴重度 finding（範本數量不一致、牌價未標佔位符）是 haiku 大概率漏掉的等級。
- **opus**：僅當審查對象本身是高風險判斷（架構決策、安全邊界）需要第二意見時。

選錯的代價不對稱：用 sonnet 審機械項多花 ~1.5 倍錢，小虧；用 haiku 審語義項會回「全部 PASS」的假陰性，直接毀掉審查的意義。**不確定時往上選一級。**

## 四、審查 prompt 的必要結構（抄 prompt-templates.md 第 5 種再加料）

實測有效的交辦結構：

1. **明說審查員身份**：「你是 fresh-context 審查員」— 讓它知道自己的無知是資產，遇到看不懂的要回報而不是跳過。
2. **檢查面向逐條列**，每條給可操作的起手式（例：「專案殘留：grep 這些詞：[詞表]」）— 面向不列，agent 只會做它擅長的那種檢查。
3. **區分「規則本體 vs 範例脈絡」**這類判定標準 — 不給標準，它會把合法的範例也報成 bug（假陽性淹沒真 finding）。
4. **要求「無發現的面向明說」**：「檢查了 X/Y/Z，無發現」和「沒檢查」是兩回事 — 不要求明說就無法區分。
5. **模擬照做 N 條規則** — 文件審查特有：光讀是不夠的，卡住的地方才是可執行性 bug。
6. **回報格式 + 總長上限 + 「不要改任何檔案」** — 審查員只回報，修是主對話的判斷（有些 finding 該駁回，例如本 session 駁回過「行數不在 90–120」— 那是驗收條件寫錯，不是內容錯）。

## 五、主對話收到 findings 之後的義務

1. 逐條判斷採納或駁回 — 審查員會有假陽性，全盤照收和全盤忽略一樣是失職。駁回要有理由（例：finding 說行數不符，實際是主對話自己的驗收估計錯了）。
2. 修完後輕量驗證（grep/wc 級別可以自己做 — 機械確認不算「大量讀取」，不必再派一輪）。
3. 向使用者回報時附數字：幾個 findings、幾高幾低、修了什麼、駁回了什麼 — 「審查通過」四個字沒有資訊量。

## 六、成本參考（本 session 實測，單次量測）

| 審查 | agent/model | tokens |
|---|---|---|
| 制度檔對抗審查（8 檔，語義）| general-purpose / sonnet | 36,554 |
| 制度檔 read-back（機械）| general-purpose / haiku | 26,973 |
| 模板包審查（9 檔，語義 + 模擬照做）| general-purpose / sonnet | 38,502 |

語義審查約 35–40k、機械 read-back 約 25–27k。審查一批文件的成本 ≈ 1.5–2 次一般派工 — 對「要移出去長期用的產出」永遠值得，對單次性的中間產物看情況省略。

## 七、從單次審查到品質迴圈（2026-07-07 補記）

本檔前六節講「一次審查怎麼派」；後來發現弱模型（Sonnet 級指揮官）缺的不是單次審查，
是**成環**：它執行完就宣稱完成，不會自發列任務清單、派完審查後只會把 findings 轉述
給使用者，不會回頭修改自己前一輪的產出。把迴圈寫成明文義務後，Sonnet 可以照跑。
完整骨架已寫成 `project/WEAK-MODEL-PROMPT-GUIDE.md`（該包的正式版本，以那份為準），
核心概念五條：

1. **計畫凍結成檔案（plan.md）**：任務清單＋每項驗收條件先落檔、使用者確認再動工。
   檔案是模型中段漂移與 /compact 之後找回任務的錨。
2. **修訂是義務不是選項**：收到 findings 後逐條處置——採納就回頭改對應產出，駁回附
   一句理由；並在 plan.md 記錄「哪條 finding 改了哪個檔」，留下修訂證據。
3. **完成的定義綁在修訂輪之後**：明文寫「階段四結束才算完成，不是執行完」。不寫，
   弱模型在執行完就收工——它對「怎樣算完成」認最後看到的明確定義。
4. **審查↔修訂最多兩輪**：第二輪後仍有 findings 就列出來問使用者，防無限迴圈燒額度。
   與派工守則「同一件事最多重試兩輪」是同一條判準。
5. **適用判準**：多產出、長期沿用、錯誤成本高於一輪審查成本（語義審查約 35–40k
   subagent tokens）才跑全迴圈；單檔小改與一次性草稿直接做＋輕量驗證。

第五節的「findings 處置義務」是這個迴圈的階段四前身；差別在於當時只要求「修完後
輕量驗證」，沒有把「回頭修改」「記錄修訂」「完成定義後移」寫成能約束弱模型的義務句。
