# Audit: can this system get Sonnet/Opus/Haiku to Fable 5-level performance?

> Written 2026-07-07, in a Fable 5 session (the user's last Fable 5 quota run).
> Answering: "without Fable 5, is this workflow good enough?" Conclusion first, evidence and gaps after.
> For how to use any of this, see `USAGE.md`.

## Conclusion (one paragraph)

**Roughly 80% of day-to-day work can hit equivalent performance — on the condition that you actually use
it the way `USAGE.md` says to, not just leave the files sitting there.** Break "Fable 5-level performance"
into pieces:

- Execution quality (implementation, search, refactoring, verification) can get close to equivalent through
  decomposition + checkable acceptance criteria + verification-isn't-self-verification.
- Staying focused across a long session can get close to equivalent through the-commander-doesn't-descend +
  context discipline.
- Three kinds of judgment — "the first cut at a vague requirement," "noticing a task's premise is wrong,"
  and "knowing when to stop" — can only be partially externalized. The ceiling there is Opus 4.8 +
  multi-option review + user arbitration.

That's not a failure of how the system is written. These three capabilities are inherently hard to reduce
to rules, and `ops/judgment-rubrics.md`#6 already honestly admits that the taste component of this can't be
fully captured.

## Item by item: Fable 5's advantages → the system's substitute → how much is recoverable

| Fable 5's advantage | The system's substitute | Recoverability |
|---|---|---|
| Stays focused over a long context | The-commander-doesn't-descend + raw material never enters the main conversation + write-as-you-go (`ops/harness-diagnosis.md`#3, #4) | High — this is the system's design sweet spot |
| Execution accuracy | Three-element delegation briefs + checkable acceptance criteria + fresh-context verification (`ops/model-dispatch.md`) | High — Sonnet's gap on well-specified tasks is already small, and the verification loop covers the rest |
| Catches its own mistakes | Change-approach signal checklist (`ops/judgment-rubrics.md`#4) + escalation carrying the failure trace | Medium — this is the weak model's biggest missing capability; the checklist is a crutch, not a prosthetic, and still needs the user to spot-check occasionally |
| First-pass decomposition of a vague requirement | Opus produces the plan first + the user front-loads more decomposition into the prompt (`USAGE.md` section 3) | Medium-low — **the biggest remaining gap**, see below |
| Taste / sensory judgment | Turning it into something testable, A/B choices handed to the user (`ops/judgment-rubrics.md`#6) | Low — the system already honestly flags this as something it can't cover |

## Four real-world failure modes of a weak commander (gaps found in this audit, and fixes)

The parts of the system written for subagents are solid. The weakest link is **the commander in the main
conversation itself**. Failure modes to expect when Sonnet is running the show:

1. **Rule drift**: partway through a session, it forgets to delegate and starts doing the work itself.
   Loading CLAUDE.md's three core rules is the first line of defense, but there's no mid-session
   re-anchoring mechanism. Fix: the user re-anchors with a trigger phrase (`USAGE.md` section 3); an
   optional reinforcement is a UserPromptSubmit hook that injects a one-line reminder — but touching
   `.claude/settings.json` is the user's call, the system won't presume to add one.
2. **Acceptance criteria written too vague**: the whole system's quality ceiling is "the acceptance criteria
   the commander writes." A weak commander produces acceptance criteria at the level of "make feature X
   work," and everything downstream turns into verification theater. Fix: for important tasks, have the
   user review the acceptance criteria before work starts (`USAGE.md` section 3's "plan first, then
   execute" phrasing) — this is the cheapest human leverage point in the entire system.
3. **Wrong direction on decomposition**: given a vague requirement, Sonnet's first cut is often wrong, and
   no amount of good execution afterward fixes running fast in the wrong direction. Fix: for vague tasks,
   run `/model opus` first to produce a plan (or a choice between two options), and let Sonnet only execute
   an already-confirmed plan (`USAGE.md` section 2, "pricier model first, cheaper model after").
4. **Failure reports get dressed up**: "should work in theory," "this ought to run fine." `ops/judgment-rubrics.md`#2
   has a rule for this, but a rule can't stop a model that doesn't read rules. The real defense is that
   verification is always fresh-context (whoever did the work doesn't get to grade their own work) — that
   already exists in the system; the key is not skipping it to save tokens.

For example: a Sonnet commander reporting "the migration script should handle the edge case now" without
having run it is exactly the pattern rule #4 is guarding against — the fix isn't a stronger claim, it's a
fresh-context agent actually running the script against the edge case.

## Cost reality (don't expect "just as good and just as fast")

Trading Fable 5 for a cheap model + this system doesn't cost you quality. It costs you **turns and
latency**: an extra delegation layer, an extra verification pass, the occasional escalate-and-redo. By API
list price (2026-06 reference values; subscription-plan conversion isn't public; Sonnet 5 list price is
UNVERIFIED): Sonnet 4.6's rate is roughly 1/3 of Fable 5's, Haiku 4.5 roughly 1/10 — even spending 50% more
turns is still a net saving. In wall-clock time, expect 1–3 extra minutes of delegation round-trips per
task. Worth it for day-to-day work; for a one-off task that needs to ship tonight at peak quality, just
force `/model opus` directly — it's faster.

- The "80%" figure is a structured estimate, not a measurement. How to verify it: pick three tasks Fable 5
  did in the past, have a Sonnet commander redo them through the `USAGE.md` workflow, and compare the
  outputs — this is the first experiment worth running for whoever picks up the next session.
- "Whether a Fable 5 request routed to Opus 4.8 by the safety classifier still burns quota": UNVERIFIED.
  The official model-config and costs docs describe the fallback mechanism but don't say how it's billed
  (verified 2026-07-07). Recommend checking the usage dashboard directly or asking Anthropic support.
- This audit was written by Fable 5 auditing a system Fable 5 itself built — it has an author's blind spot.
  A separate fresh-context Sonnet was dispatched as an adversarial reviewer (2026-07-07), producing 6
  findings:
  - 3 high: the index omitted the handoff letter, a live file still had leftover `{{}}` template slots,
    USAGE's opening was missing the alias warning.
  - 2 medium: the delegation template's subject was ambiguous, the environment section wasn't instantiated.
  - 1 low: a dangling reference.

  All of them were adopted and fixed. The list of unmeasured numbers lives in
  `meta/letter-to-future-sessions.md`.
