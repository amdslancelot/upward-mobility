# The Brief and the Plan (self-applied by the model)

> Reader: the main-thread model. This covers phase 0 (brief) and phase 1 (plan) of a large task — turning it into a frozen `plan.md` before any execution starts. Execution follows the workflow at the end of this file (you do it yourself, in this context); review and escalation are their own skills' jobs — see "Once plan.md is frozen" below.
> Use this for tasks with multiple deliverables, half a day or more of work, or output meant to stick around long-term. Skip it for a small single-file edit or a one-off draft — the fixed overhead of a brief and a plan eats the benefit; do the work plus a lightweight check instead (output that will be kept still gets a fresh-context review; one-off intermediates may skip it).

## plan.md template (create before starting work, maintain throughout)

```
# <task name> — plan.md
Status: <planning / executing: item N / reviewing / revising>   Updated: <date>

## Brief (pinned down before work starts; this section stands in for a senior model's "get the first cut right")
- Goal and motivation: <what to do, why>
- Acceptance criteria (decidable, itemized): <"following these steps gets tests green" is fine; "written clearly" is not>
- Assumptions: <things you're taking as true in order to proceed — if something breaks, check this first for a wrong assumption. Any assumption about an external format, API, protocol, or third-party behavior must be verified against a primary artifact (a real sample file, actual command output, or official documentation — blogs and community posts don't count) before the plan freezes; one you can't verify stays labeled unverified under Risks, and the code that depends on it must tolerate variance in it>
- Risks / out of scope: <known risks, explicitly excluded scope>
- Review-effort level: <LOW (default) | MED | HIGH — the level the end-of-run consumer-seat review runs at; only a level the user actually requested may appear here, and absent a user request write LOW>

## Task list
- [ ] 1. <task name>
  - Target: <files/scope to touch>   Product: <what it emits: file, diff, report>
  - Dependency: <task #s that must finish first, or —>
  - Acceptance / Review: <decidable done-criterion; the main thread runs the gate once when the item finishes and logs the evidence; no per-item verifier agent; the end-of-run consumer-seat reviewer is the only independent pass>
- [ ] 2. ...
- [ ] N. Consumer-seat reality check (always the last item; executed by the fresh-context consumer-seat reviewer — never by the plan's author; see "Ground the criteria outside the run" below)
  - Acceptance: <phrased from the consumer's seat: "run deploy.sh and traffic reaches the app," "boot the production build and a real request round-trips" — never the builder's "build green" or "selfcheck passes against the fixture">

## Checkpoints (log one every time a milestone passes its acceptance criteria)
- <milestone> → <git commit: sha or one-sentence description>

## Revision log (fill in after review)
- <finding> → fixed at <file:line> | or rejected: <one-sentence reason>
```

## The two phases this skill owns

**Phase 0 — brief (before starting work)**: write the goal, acceptance criteria, assumptions, and risks into the top of plan.md. If the direction is uncertain (requirements are vague, or you're guessing at the user's preferences or business context) → stop and confirm before starting work; see `upward-ops-judge` skill#2 for the criteria. Don't skip this step: flawless execution in the wrong direction is the most expensive failure there is.

**Phase 1 — plan**: break the task into a list, attach decidable acceptance criteria to each item, scan the "Known killers" section below and copy each matching line into the relevant item's criteria, then freeze it into plan.md. If you drift mid-task or hit a `/compact`, this file is the anchor that gets you back to your task list.

## Ground the criteria outside the run

You are writing the exam you will later be graded on: any misunderstanding you hold now gets baked into the acceptance criteria, and every later review will verify the work against that misunderstanding instead of catching it. Three rules keep the criteria anchored to reality rather than to your own beliefs:

1. **External evidence rule**: every load-bearing assumption about an external format, API, protocol, or third-party behavior gets verified against a primary artifact — a real sample file, actual command output, or official documentation fetched now — before the plan freezes. Never from memory, and never from secondary sources: blogs, forum threads, and third-party tool sites can suggest what a format looks like, but they can never upgrade an assumption to "verified." An assumption resting only on secondary sources stays labeled unverified. Can't reach a primary artifact? The assumption goes under Risks as unverified, and rule 2 governs the code built on it. Do this fetching through a read-only research dispatch (template #2 in `upward-ops-dispatch`; load that skill first) — the main context never calls WebSearch or WebFetch itself; a run that was merely told to "prefer" dispatching carried its inline fetches to the end and paid a third of its total tokens re-reading them. The verification is just as real provided the digest cites the primary source per template #2's source requirements (an item the dispatch reports UNVERIFIED stays unverified). When a fixture will need the sample itself (rule 3), have the dispatch save the fetched sample to a scratch path and report the path.
2. **Unverified means tolerant**: code that depends on an assumption still unverified at plan freeze must tolerate variance in that detail — parse defensively, accept extra or missing fields, degrade with a clear warning that names the assumption — never hard-fail on an exact match of the unverified detail. A strict validator encoding an unverified belief turns "our guess was slightly off" into total failure on first real use; a wrong belief with a "verified" stamp fails harder than a wrong belief alone.
3. **Fixture provenance rule**: a test fixture you author from the same belief as the code can only confirm the belief, never test it. Any fixture that validates an external-format assumption must derive from a real external sample. No real sample obtainable in principle (the source is an arbitrary end user, not a nameable system)? Ask the user for a sample when a user is available; otherwise the fixture gets authored from the best evidence at hand, the assumption stays labeled unverified, and rule 2 still applies to the code. A self-authored fixture is a build gate — it never counts as consumer-seat evidence and never satisfies rule 4.
4. **Mandatory final task**: every plan ends with a consumer-seat reality check — run the finished artifact the way its real consumer would and watch it work. Its criterion is phrased from the consumer's seat ("run `deploy.sh` and traffic reaches the app," "upload a real export and rows appear"), never the builder's ("build green," "selfcheck passes against the fixture"). When the real consumer path is infeasible in the current environment (no credentials, no cloud account), step down this ladder and take the highest rung that runs: real deploy → boot the built artifact locally and exercise its real entrypoints (real HTTP requests against the production build from a fresh state, a rendered manifest, a real CLI invocation) → if nothing is runnable at all, report the check as NOT DONE rather than substituting a fixture selfcheck. Whatever rung runs: exercise any repeatable consumer path **twice** — a re-import, a scheduled sync, a redeploy — and require the second run to change nothing (no duplicated rows, no re-billed API calls, no reverted config); "works once, corrupts on repeat" has shipped through single-run checks twice. This task is executed by the fresh-context consumer-seat reviewer in the execution workflow below, never by the plan's author — and it runs at the review-effort level the run's brief declares (the level definitions live in `upward-ops-dispatch`; the default is LOW). At LOW the runnable rungs of the ladder are not attempted: the reviewer performs a static Known-killers sweep plus the criteria attack instead, and this item is checked off carrying its report's STATIC-ONLY marker — a third state distinct from a consumer-seat run and from NOT DONE, recording that the boot-and-exercise half was deliberately not purchased. The boot-and-exercise floor above, including the run-twice requirement, applies when the declared level is MED or HIGH.

**Good**: the plan assumes Takeout "Saved" exports are GeoJSON → before freezing, have a research dispatch fetch a real Takeout export and report what it actually contains. (They're CSV — the plan just avoided shipping a parser for a format that never arrives.) ✅
**Bad**: write a GeoJSON parser, author a GeoJSON fixture from the same belief, set the criterion to "selfcheck passes against the fixture." Every box goes green; a real export imports nothing. ❌
**Bad**: "verified the CSV header via web search" against blogs and forum posts, then a parser that hard-throws on any other header. The sources were wrong, the "verified" label justified strictness, and a parser that would have mostly worked became one that rejects every real export. ❌

## Known killers (write the matching line into the relevant item's acceptance criteria)

These defect classes have shipped repeatedly across measured builds of unrelated tasks — each one passed every build gate and died in production state no gate reaches. When the plan touches the matching area, copy the line into that item's acceptance criteria so the gate and the reviewer both check it:

- **Secrets and substituted config never land in git-tracked files.** Anything a redeploy's `git reset`/`git pull` touches reverts to the placeholder; substitute at apply time from untracked state (an env file outside the repo), never by editing tracked files in place.
- **Degraded, fallback, or error results are never written to a permanent cache.** A cache with a "never fetch twice" guarantee must store only confirmed-good results; caching a failure or a missing-API-key fallback makes the misconfiguration permanent.
- **Secret and token comparisons use a constant-time compare** (`crypto.timingSafeEqual` or the platform equivalent) — never `==`/`===`/`!==` on the raw strings. Raw-string comparisons have shipped in twelve of twelve measured builds.
- **Opening a port means opening it at every layer it crosses**: the cloud ACL *and* the host firewall of the actual OS image (cloud Ubuntu images commonly ship a default-REJECT iptables set). A green `terraform apply` with an unreachable service has shipped twice.
- **Auth-framework actions go through the framework's own helper, never a raw form post.** A hand-written `<form action="/api/auth/signout">` bypasses the framework's CSRF token and the framework rejects every submission (Auth.js v5 throws `MissingCSRF`); a sign-out that fails 100% of the time shipped through a build whose gates only exercised sign-in.
- **A webhook-triggered deploy takes an exclusive lock that serializes or re-triggers — it never silently drops.** Every step that touches the working tree, the `git reset` included, happens inside the lock, so two pushes in quick succession cannot interleave a reset with a still-running build and produce a mixed-tree image; and a push that finds the lock held must either wait for it or re-trigger the deploy after the holder finishes — a `flock -n` that just exits leaves the newest commit silently undeployed.
- **Every path that creates a user account goes through the one provisioning code path.** Interactive sign-in, a scheduled or cron-driven import, a background sync — all of them. A parallel scheme that mints synthetic user ids for unattended paths collides with the real identity later: a measured build's cron provisioned a synthetic id on its first unattended run that permanently bricked the owner's own first interactive sign-in.
- **Uploads stream to disk with an explicit size cap.** Never read a whole uploaded file into memory (`arrayBuffer()`, an in-memory unzip) on a pod or VM with a memory limit — one oversized upload becomes an out-of-memory crash of the whole service instead of a rejected request.

## Once plan.md is frozen: the execution workflow

Planning stops here. The rest of the run follows the loop below. The Skill-tool calls in it are mandatory steps of the workflow, not reading suggestions — a prose mention of a skill loads nothing; only calling the Skill tool puts its rules in front of you, and a run that skips these calls executes with the safeguards missing.

1. **Execute the items yourself, in this context.** Work through items 1 to N−1 in dependency order (item N, the consumer-seat reality check, is discharged by step 4's dispatch, not here); when an item finishes, run its acceptance gate once, log the evidence as a checkpoint in plan.md, and check the item off. Staying in the warm context is the measured cheap path — a run that dispatched every item to fresh subagents paid more in cold-starts (each subagent re-reading the plan and the tree) than it saved, and shipped lower-quality code because a cheaper model wrote it. Work with the token formula in view: the bill is calls × context, and everything added here is re-read by every later call. So execute each item as design → one batch → gate: decide what all of the item's files contain, emit every one of them in a single message (one API call carrying several Write calls), then run the item's gate after the batch — never one file per call; a Write or Edit travels alone only when it depends on the result of the immediately preceding call (a fix answering a failed gate, an edit that needed a Read first) — or when it is the only call the item requires: a batch of one is still one message. Bookkeeping is never a whole call: a message whose only tool calls are plan.md Edits (a check-off, a checkpoint line) is a rule violation — hold the edit and emit it inside the next message you were going to send anyway (the next item's first batch or gate; the step-4 Skill-tool call; in step 5, the fix batch or re-run gate it records). The one exception is the run's terminal plan.md update, after which no message follows at all — that is the batch-of-one the batching rule already permits. A run that ticked its checkboxes in standalone calls paid ~90K of context re-reading per tick, six times. A run that ignored this wrote one file per call and spent 69% of its entire bill re-reading its own growing context between single-file writes. Pipe a verbose gate's output to a file, judge pass/fail by the exit code, and read back only the tail as the logged evidence while it passes (open the full log only when it fails — a failure must be seen in full), and skip conversational task-tracker tools in a headless run: plan.md is the task list.
2. **Subagents are read-only in this workflow.** Dispatch one only to read, run, and report — bulk material that would flood this context, a read-back, a second opinion, and the step-4 reviewer that boots and exercises the artifact — never to build: implementation and fixes always happen here, in the main context. A multi-layer build is not "parallel workstreams" — layers that share a contract must share a context; a measured run that split three "independent" layers across three builders shipped its fatal defects exactly at the seams between them while costing more than serial work in one context. Bulk is measured in time carried, not just size — there is no size threshold, and the working set (plan.md, the files under edit) is exempt: what the rule targets is research or reference material consulted for a few items but held to the end of the run, which compounds into a flood — one measured run paid a third of its total tokens re-reading 45K of early web research it no longer needed. Fetch such material through a reader dispatch that returns a short digest, and carry only the digest. Before the first dispatch of any kind, load `upward-ops-dispatch` via the Skill tool and follow its brief template (a plan-phase research dispatch counts as the first dispatch and may already have loaded it). Dispatch in the foreground — every background dispatch wakes this thread again with a paid notification round.
3. **When an item's gate fails twice, a fix won't take, or something breaks for an unknown reason** → load `upward-debug` via the Skill tool and follow its signal-first loop. **When stuck on direction, weighing a model escalation, or facing a call only the user can make** → load `upward-ops-judge` via the Skill tool.
4. **After the last main-thread item (N−1), load `upward-ops-review` via the Skill tool — and load `upward-ops-dispatch` too if this run hasn't loaded it yet** — then dispatch exactly one fresh-context consumer-seat review (foreground) per the review-dispatch section of `upward-ops-dispatch`, at the review-effort level the run's brief declares (default LOW). This single dispatch does two jobs at once: it executes the consumer-seat reality check (task N of the plan) — at MED or HIGH by booting and exercising the artifact per rule 4's ladder, including running any repeatable consumer path twice; at LOW by the static Known-killers sweep, with the report carrying the STATIC-ONLY marker — and it attacks the acceptance criteria against the original request, which travels verbatim in its prompt. It does not re-run install/build/test gates you already ran and logged with execution evidence; its scope is the real entrypoints and the blind spots, not your checklist. You never perform this review yourself — authoring the criteria counts as doing the work, and the author verifying against their own criteria is the Goodhart loop this workflow exists to break. When the review comes back, check item N off on the reviewer's reported evidence (at LOW, the check-off carries the STATIC-ONLY marker); a NOT DONE consumer-seat report is itself a finding — take it into step 5.
5. **When findings come back**: triage each one per `upward-ops-review`, apply accepted fixes yourself in this context, re-run only the specific gate each finding names, log the revision, and commit the checkpoint.

Save progress in plan.md as you go — one line per checkpoint, and a revision log that records finding → fix, not narrative. If the session is interactive, tell the user `plan.md` is frozen and wait for a go before executing item 1; in an autonomous run with no user to ask, proceed directly into the workflow above.
